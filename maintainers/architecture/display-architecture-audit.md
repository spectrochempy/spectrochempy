# Display Architecture Audit

## Status

Deferred architecture audit.

No display refactor is currently planned. This document records the current
fragmentation and provides a starting point for a future display contract.

## Executive Summary

SpectroChemPy does not currently have a unified display architecture.

The display layer is functional, but representation semantics are fragmented
across several parallel paths:

- compact text representations built from `__repr__()` helpers;
- detailed text representations built from `_cstr()` helpers;
- HTML representations generated by parsing `_cstr()` output through
  `convert_to_html()`;
- `Project` HTML representation generated by transforming `__str__()` output
  directly.

The main issue is not visual inconsistency. It is the absence of one semantic
source of truth for what an object should communicate to users.

## Current Display Paths

### Coord

Major entry points:

- `Coord.__repr__()`;
- `Coord.__str__()`;
- inherited `_repr_html_()`;
- `Coord._cstr()`;
- inherited `_repr_value()`, `_str_shape()`, and `_str_value()`.

Compact text uses low-level `NDArray` helpers. Detailed text uses `_cstr()`.
HTML uses `convert_to_html(self)`, which calls `_cstr()`.

### CoordSet

Major entry points:

- `CoordSet.__repr__()`;
- `CoordSet.__str__()`;
- `CoordSet._repr_html_()`;
- `CoordSet._cstr()`.

Compact text is assembled directly in `CoordSet.__repr__()`. Detailed text is
assembled separately in `_cstr()`. HTML again depends on `_cstr()`.

### NDDataset

Major entry points:

- inherited `NDArray.__repr__()`;
- inherited `NDArray.__str__()`;
- inherited `_repr_html_()`;
- `NDDataset._cstr()`;
- `NDDataset._str_dims()`.

Compact text uses `NDArray` helpers. Detailed text uses `NDDataset._cstr()`,
with coordinate detail delegated to `CoordSet._cstr()`. HTML parses `_cstr()`
output.

### Project

Major entry points:

- `Project.__str__()`;
- `Project._repr_html_()`;
- default Python `repr`.

`Project` does not participate in the shared `convert_to_html()` / `_cstr()`
path and has no project-specific `__repr__()`.

## Sources of Truth

There is no single source of truth for display semantics.

Compact text for array-like objects is governed by low-level `NDArray` helpers.
Detailed text is governed by object-specific `_cstr()` methods. HTML display
for `Coord`, `CoordSet`, and `NDDataset` is derived by parsing `_cstr()` output
with sentinel markers and regular expressions.

For `Project`, HTML display has a separate source of truth: `Project.__str__()`
plus direct string substitution.

## Implicit Display Semantics

SpectroChemPy already has an implicit display contract.

Compact text generally shows:

- object type;
- dtype or empty state;
- units or unitless state;
- shape or size.

Detailed `NDDataset` display treats the following as essential:

- identity;
- authorship;
- creation and modification timestamps;
- description;
- history;
- data summary;
- shape;
- dimensions and coordinates.

Detailed `Coord` and `CoordSet` display treats coordinate values, labels,
dimension grouping, titles, and references as essential.

## Architectural Risks

### Path-Dependent Semantics

What an object communicates depends heavily on whether the caller uses
`repr()`, `str()`, `_cstr()`, `pstr()`, or `_repr_html_()`.

### HTML Depends on Text Parsing

`convert_to_html()` consumes formatted text with sentinel markers such as
`\0`, `\0\0`, and section headings. Semantic content and formatting protocol
are mixed in one string channel.

### Project Is Separate

`Project` has an independent representation path. This weakens consistency
across core object families.

### Duplication Raises Review Cost

Changing what should be shown may require touching compact repr helpers,
`_cstr()` helpers, HTML parsing expectations, and object-specific branches.

### Test Coverage Is Partial

Representation behavior is tested locally, but there is no single
characterization suite mapping display semantics across `Coord`, `CoordSet`,
`NDDataset`, and `Project`.

## Potential Future Direction

A future Display Contract could define:

- what each object must communicate in compact text;
- what each object must communicate in detailed text;
- what each object must communicate in notebook / HTML display;
- which fields are essential, optional, or intentionally hidden.

The value would be semantic rather than cosmetic:

- separate representation semantics from formatting mechanics;
- reduce path-dependent display behavior;
- provide one reference for display reviews.
