# modeldata — Semantic Contract RFC

**Status:** Draft / Maintainer discussion document

**Implementation status:** NDDataset.modeldata has been removed following the
analysis below.  See `#1168`_ and commit logs for details.

## Purpose

This document answers two questions:

1. What is `modeldata`?
2. What semantic contract should govern it?

It is an architectural analysis only.  No implementation changes, no migration
plans, no deprecation proposals.

The goal is to transform `modeldata` from an implicit historical field into a
documented architectural concept before any future redesign.

See also upstream issue `#1168`_ for the original discussion context.

.. _#1168: https://github.com/spectrochempy/spectrochempy/issues/1168

---

## Historical Context

### Origin

`modeldata` has been present on `NDDataset` since at least v0.2.0 (January
2021, commit `f0eefb8a53`).  Its trait declaration carries the comment:

```python
# model data (e.g., for fit)
_modeldata = Array(tr.Float(), allow_none=True)
```

The property docstring reads:

> Data eventually generated by modelling of the data.

The original use case appears to be spectroscopic curve fitting: storing
the synthesised model components computed by a fitting routine so they can
be overlaid on the original data during visualisation.

### Original Workflow

The `Optimize` class (`optimize.py`) contains a `_get_modeldata` method that
builds a local numpy array of model component curves:

- Row 0: baseline
- Rows 1..n: individual model components (peaks, bands)
- Last row: sum of all components

This array is shaped `(n_components + 2, x_size)` — always **1D along the
component axis**, 1D along the spectral coordinate.

**Critical observation:** `_get_modeldata` returns the array to the caller but
**never stores it back** to `dataset.modeldata`.  The `Optimize` class has its
own separate `modeldata` trait (line 114) that is unrelated to the
`NDDataset.modeldata` property.

This means the `NDDataset.modeldata` property is **never written by any
production code in the current codebase**.

## Current Usage Audit

### Definition

| Aspect | Detail |
|--------|--------|
| **Defined on** | `NDDataset` only |
| **Storage** | `_modeldata = Array(tr.Float(), allow_none=True)` |
| **Default** | `None` |
| **Property** | Simple getter/setter (no validation, no shape check, no transformation) |
| **Copy mechanism** | Listed in `_attributes_()` → deep-copied verbatim on every `copy()` |
| **Equality** | Explicitly excluded from `__eq__` |

### Assignment

| Location | Count | Details |
|----------|-------|---------|
| Production `src/` | **0** | No `.modeldata =` assignment anywhere |
| Test files | 11 | Across 4 baseline test files — always `np.full(...)` to fixed shape |

### Reading

| Location | Count | Details |
|----------|-------|---------|
| Production `src/` | **1** | `plot1d.py:385` → reads `new.modeldata` for `plot_model=True` option |
| Test files | ~36 | Assertions verifying stale shape, preserved values |

### Modification

No production code modifies `modeldata`.  Tests only set it, never modify in
place.

### Classes Involved

- `NDDataset` — the only class with a `modeldata` property
- `Optimize` — has its own `modeldata` trait (unrelated, on `Optimize`, not on
  `NDDataset`)
- `plot1d` — reads the property for overlay plotting

### Is It Actively Used?

**Minimally.**  The property exists, deep-copies through every operation, and
is read by exactly one production code path (`plot1d` with `plot_model=True`).
No production code ever writes to it.

`modeldata` is effectively **latent infrastructure** — it is carried forward
by every dataset operation but never populated or consumed by the rest of the
framework.

## Conceptual Classification

### Candidate Categories

| Category | Fit? | Reasoning |
|----------|------|-----------|
| **Scientific Context** | No | Scientific context (title, name, author, description, origin, meta) survives geometry changes unconditionally.  `modeldata` is geometry-dependent — it refers to the coordinate grid. |
| **Provenance** | No | Provenance records where data came from.  `modeldata` is a computation result, not a record of origin. |
| **Structural Information** | **Yes** | Like CoordSet, labels, masks, and ROI, `modeldata` depends on coordinate geometry and data shape.  It becomes stale when either changes. |
| **Analysis State** | Partial | It stores analysis output (model components) but is never populated by analysis code.  The `Optimize` class built its own independent `modeldata` array. |
| **Fitting State** | Partial | The name and original comment suggest fitting, but the property is decoupled from the actual fit machinery. |

### Assessment

`modeldata` is **structural information** in terms of the RFC's semantic
building blocks.  It belongs in the same category as CoordSet, labels, masks,
and ROI: its validity depends on geometry, domain, shape, or representation.

However, it differs from CoordSet and masks in a critical way:

- CoordSet: rebuilt when geometry changes
- Masks: reconstructed when geometry changes
- Labels: carried over on exact match only
- ROI: preserved verbatim (stale risk)
- **`modeldata`: preserved verbatim (stale risk)**

In current behaviour, `modeldata` is treated as if it were scientific context
(it is blindly deep-copied), but its semantics require it to be structural
(it only makes sense relative to a specific coordinate grid and data shape).

## Relationship to Scientific Context, Provenance, and Structural Information

`modeldata` straddles a conceptual boundary:

| Perspective | Implication |
|-------------|-------------|
| **As scientific context** | Should survive everything.  Current copy behaviour matches this, but the semantics don't — `modeldata` is not a free-text description. |
| **As structural information** | Should be recomputed or dropped when geometry changes.  Current behaviour violates this — it is preserved verbatim, producing stale data. |
| **As provenance** | Unconvincing.  `modeldata` is a computed result, not a record. |
| **As analysis state** | Plausible.  It is the output of a modelling operation stored alongside the input data for comparison. |

The current implementation accidentally treats it as scientific context (copy
verbatim).  The semantics require it to be structural (geometry-dependent).

---

## Identity Implications

Using the RFC's identity framework:

| Identity class | Should modeldata survive? | Current behaviour |
|---------------|--------------------------|-------------------|
| **Same scientific object** (arithmetic, ufuncs) | **Yes** — model values remain valid for the same data | ✅ Preserved verbatim |
| **Same object, changed representation** (interpolation, reshape) | **No** — model is tied to the original grid | ❌ Preserved verbatim (stale) |
| **Derived scientific object** (analysis outputs) | **Maybe** — depends on whether the model is part of the input or the output | Inconsistent: Group A drops, Group B preserves |
| **Multi-source synthesized object** (concatenate, stack) | **No** — which source's model survives is accidental | ❌ Preserved from last source (accidental) |

**Conclusion:** The current behaviour is defensible only for identity-preserving
operations.  For representation changes, derived objects, and multi-source
results, the current verbatim copy is a stale-field bug.

## Dimensionality Analysis

### Current implementation

`_modeldata = Array(tr.Float(), allow_none=True)` — no shape constraint.
The setter performs no validation.  Any n-dimensional array is accepted.

### Observed usage

- Tests set `modeldata = np.full((5, 7), val)` — matching the dataset's shape.
- `Optimize._get_modeldata` builds a 2D array with shape `(n_components + 2, x_size)`.
  This is **not** the same shape as the dataset — it has a different first
  dimension.

### Is nD meaningfully supported?

The property technically accepts any shape, but:

- No production code validates shape compatibility.
- The only production reader (`plot1d`) accesses `.T` on the modeldata array,
  assuming it is 2D.
- The `Optimize` output shape differs from the dataset shape, so assigning it
  back to `dataset.modeldata` would produce a shape mismatch if any operation
  expected alignment.

**Assessment:** nD is accidentally permitted, not meaningfully supported.
The property's lack of shape validation means a mismatched `modeldata` can
silently coexist with the data it supposedly models.

### Meaningful shapes

| Shape | Semantics | Status |
|-------|-----------|--------|
| Same as dataset | Model of each data point | Most natural interpretation |
| (n_components, x_size) | Component decomposition | Used by `Optimize` internally |
| Scalar | Single model parameter | Never observed |
| None | No model data | Default |

## Relationship to Fit / Optimisation

### Current state

The `Optimize` class:

1. Has its **own** `modeldata` trait (separate from `NDDataset.modeldata`)
2. Builds a local `modeldata` numpy array in `_get_modeldata`
3. Returns it to the caller
4. **Never** stores it in `dataset.modeldata`

The `NDDataset.modeldata` property is therefore **orphaned** — it is defined
on the data class, but the fitting machinery that should produce meaningful
values does not use it.

### Central question

> Is `modeldata` a property of data, or a property of a model explaining data?

- If it is a **property of data**, it should survive data transformations
  (be sliced with data, interpolated with data, etc.) — but then it should
  be transformed consistently, not preserved verbatim.
- If it is a **property of a model**, it belongs on the fit/optimisation
  result object, not on the dataset.  The dataset may reference the model
  result, but should not own it.

The current implementation implicitly assumes the first interpretation
(carried on data), while the original comment and the `Optimize` machinery
implicitly assume the second (model explaining data).

This is the root contradiction.

## Architectural Options

### Option A: Keep modeldata on NDDataset, define a clear propagation contract

- Define which operations preserve, recompute, or drop `modeldata`.
- Add shape validation to the setter.
- Make `modeldata` follow the same rules as other structural information.

**Advantages:** Minimal change; backward-compatible for valid usage.
**Drawbacks:** Does not resolve the orphaned-property problem (nothing writes it).
**Compatibility:** Existing tests would need updated assertions.

### Option B: Restrict modeldata to identity-preserving operations

- `modeldata` survives arithmetic, ufuncs, and elementwise processing.
- It is dropped (set to `None`) for representation changes, derived objects,
  and multi-source results.

**Advantages:** Eliminates stale-field risk; clear semantic boundary.
**Drawbacks:** Silent data loss for users who set `modeldata` and then
interpolate/reshape.
**Compatibility:** Breaking change for existing workflows that set `modeldata`
and rely on it surviving geometry changes.

### Option C: Move modeldata toward fit-owned state

- Remove `modeldata` from `NDDataset`.
- `Optimize` (or a future fit result object) owns the model data.
- The dataset can optionally reference a fit result.

**Advantages:** Resolves the orphaned-property problem; conceptually clean.
**Drawbacks:** Major API change; breaks all existing `modeldata` access.
**Compatibility:** High breakage — code and tests that set/read `modeldata`
would need updating.

### Option D: Separate model objects from datasets

- Introduce a `ModelResult` or `FitResult` class.
- `Optimize` returns a `ModelResult` instead of mutating datasets.
- `modeldata` lives on `ModelResult`.
- Datasets lose the `modeldata` property entirely.

**Advantages:** Architecturally clean; separates concerns.
**Drawbacks:** Requires new class, new API surface, migration path.
**Compatibility:** Maximum breakage; requires deprecation cycle.

### Option E: modeldata as visualisation hint (documented narrowing)

- Acknowledge that `modeldata`'s only production use is plotting.
- Narrow its documented semantics to *stored visualisation overlay*.
- Define: "modeldata is an array that, if set and shape-compatible, may be
  plotted alongside the dataset.  No other operation reads or transforms it."

**Advantages:** Matches actual usage; honest about current state.
**Drawbacks:** Does not resolve the stale-field problem; postpones the
architectural decision.
**Compatibility:** Zero breakage; purely documentary.

## Maintainer Assessment

### What modeldata appears to be today

`modeldata` is **latent infrastructure**.  It is defined, deep-copied, and
protected, but never meaningfully populated by production code.  Its only
production read is a visualisation overlay path controlled by
`plot_model=True`.

### What category it most naturally belongs to

**Structural information** — like CoordSet, masks, labels, and ROI, its
validity depends on coordinate geometry and data shape.  It is not scientific
context, not provenance, and not analysis state (because nothing populates it
as such).

### Whether its current propagation behaviour is defensible

**No.**  The current verbatim deep-copy through every operation is defensible
only because nothing writes to `modeldata` in practice.  If a user did set it,
it would silently become stale after any geometry-changing operation.  The
behaviour is accidental, not intentional.

### Whether a future implementation RFC is justified

**Yes**, but only once a semantic direction is chosen.  The three dimensions
to decide are:

1. **Ownership** — does `modeldata` belong on `NDDataset` or on a fit result?
2. **Propagation** — should it survive all operations, geometry-preserving
   only, or none?
3. **Validation** — should the setter enforce shape compatibility?

The minimal safe next step is **Option E** (document the current narrow usage
as a visualisation hint), followed by a decision on ownership before any
implementation work.

### Summary

| Question | Answer |
|----------|--------|
| What is it? | A latently defined structural field carried on `NDDataset` |
| What was it for? | Storing fit model components for visualisation overlay |
| Does anything write it? | No production code |
| Does anything read it? | One plotting code path |
| Is current propagation correct? | Only by accident (nothing sets it) |
| Should it be redesigned? | Yes — but semantic direction first |
| Recommended next step | Option E (document) then Option C or D (ownership) |

## Maintainer Decision

Following the audit above, the maintainer has decided that
NDDataset.modeldata is **orphaned historical infrastructure** and will be
removed rather than given a new propagation contract.

### Rationale

1. **No production writes.**  Zero production code paths write to
   `NDDataset.modeldata`.
2. **Single production reader.**  The only production read was `plot1d` with
   `plot_model=True`, which has been updated to emit a `FutureWarning` instead
   of accessing the removed attribute.
3. **Accidental propagation.**  `modeldata` was deep-copied through every
   dataset operation because it was listed in `_attributes_()`, but this
   propagation was never intentional and always produced stale structural state.
4. **Optimize already separated.**  The `Optimize` class has its own
   `modeldata` trait and `_get_modeldata` method that operate independently of
   `NDDataset.modeldata`.  This separation is correct and is preserved.
5. **Fit/model outputs should be explicit.**  Rather than storing hidden model
   state on `NDDataset`, fit and modelling results should be represented as
   explicit `NDDataset` objects or dedicated fit-result objects.

### What was done

- Removed `_modeldata` trait, default, property getter/setter from `NDDataset`.
- Removed `modeldata` from `_attributes_()` and from the `__eq__` exclusion list.
- Updated `plot1d(plot_model=True)` to emit a `FutureWarning` explaining the
  removal instead of silently reading orphaned dataset state.
- Removed all test setup and assertions that set or read `NDDataset.modeldata`.
- Updated the `testing.py` utility to remove `modeldata` from its comparison
  exclusion list.
- Updated the Metadata Contract RFC (`metadata-contract.md`) to reflect that
  `modeldata` is no longer an `NDDataset` field.
- Added changelog entry documenting the breaking change.
- Closes `#1168`_.

### Not changed (out of scope)

- `Optimize.modeldata` — left intact; the `Optimize` class manages its own
  model data independently.
- Plotting API — no new `ModelResult` or `FitResult` class introduced; no new
  plotting protocol added.
